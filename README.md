# WoW Crucible

The complete local-tool replacement contract is tracked in [`docs/TOOL-CONSOLIDATION-MATRIX.md`](docs/TOOL-CONSOLIDATION-MATRIX.md). Legacy tools are behavioral and format references during development; no worthwhile workflow is intentionally left as a permanent external dependency.

> [!WARNING]
> WoW Crucible is in very early development. Back up your files and validate generated client/server changes before using them on a live project.

WoW Crucible is an open-source World of Warcraft content-authoring toolkit. Its primary verified target is 3.3.5a (build 12340), with an extensible target-profile system for Classic 1.12.1, TBC 2.4.3, experimental Cata 4.3.4, and future community targets. The long-term goal is one coherent workflow for DBC/DB2 editing, patch creation, spells, items, creatures, races, classes, and current server-core integration.

Client formats and server integration are separate: a client-build profile declares file/archive capabilities, while a live schema adapter targets what the selected server actually exposes. The verified server contract targets current AzerothCore `master` and current TrinityCore branch `3.3.5`. See [docs/COMPATIBILITY.md](docs/COMPATIBILITY.md).

- Selects built-in target profiles for Classic 5875, TBC 8606, WotLK 12340, and experimental Cata 15595; accepts external JSON profiles without recompilation.
- Connects to a live MySQL/MariaDB world database, keeps the password in memory only, and inspects actual content-table capabilities before enabling server writes.

Keeper staging re-verifies hashes and produces a tiny manifest-driven patch tree without changing the processed library.
- Resolves model dependency closure before accepting a deployable keeper. Same-source SKIN, ANIM, and embedded BLP paths are included automatically; missing paths and cross-source-only matches block staging, while replaceable character/creature texture slots are reported as explicit future DBC/appearance bindings.
- Creates portable content-project folders with separate Assets, DBC, SQL, Manifests, Reports, and Staging outputs plus a persistent ID registry. The same-window **Projects & shared IDs** workspace creates/opens projects, selects target profiles and asset libraries, shows every authoritative occupancy source and reservation, and scans every mapped live SQL identity plus matching DBC stable key before allocation. It refuses incomplete coverage, enforces WotLK's 31-ID race/class ceiling, shares one collision namespace between mount and spell IDs, and accepts explicit occupied lists for custom domains. Items, spells/mount drafts, creatures, gameobjects, quests, playable classes, and playable races use the shared registry. The native class bundle clones `ChrClasses`, playable race/outfit rows, three class-access DBC masks, and recognized complete SQL starting/stat rows. The native race bundle clones the full 16-table client surface—`ChrRaces`, class pairs, outfits, appearance sections, hair/facial/barber choices, names, vocal/emote data, faction reputation and skill/talent masks—plus recognized complete SQL rows. A race plan can bind reviewed male/female displays already present in the authoritative DBCs or promote both complete source-layer appearance chains in one coordinated pass. Promotion semantically reuses equal rows, collision-remaps genuinely different occupied model/display/extra/item-display identities, rewrites proven references, and binds the resulting target displays into `ChrRaces`. An optional source customization race now replaces the baseline `CharSections`, hair-geoset/texture, facial-hair, and barber clone with that source layer's exact rows. Physical IDs are semantically reused or collision-remapped, generated-key facial-hair rows remain append-only, missing source surfaces stay explicit, and every referenced `CharSections` texture joins the strict same-provenance asset closure. Missing assets, invalid model graphs, stale source/target DBCs, unexplained target-race rows, different-byte path ambiguity, or relevant live `*_dbc` overlay rows block the build; Crucible will not let SQL silently override or mix a promoted customization surface. Both bundle providers bind exact source hashes/rows/schema fingerprints, reject occupied target identities (including non-player race IDs), and build only a new DBC/SQL/manifest/tiny-MPQ/receipt folder. They never change the live database, server, or client and explicitly report remaining core/UI/spell/asset work.

Model visibility is path-aware by default: playable race/sex bodies use mutually exclusive character filtering, while direct item, prop, map, and generic M2 loads retain the complete SKIN. Any explicit DBC/manual selection also enters character-filtered mode, so generic assets no longer lose valid sections merely because the renderer's old global default was designed for player bodies.

Client indexing preserves every still-hash-only entry under its synthetic StormLib name, reports the unresolved count, and can retry name recovery from a corpus built across multiple clients. `--resolved-only` prevents synthetic names from being mistaken for reusable folder paths; `--anonymous-only` quarantines those unresolved payloads for signature/content inspection.

Standalone MPQ listing and extraction also accept `--listfile=paths.txt`; unresolved `File000...` placeholders are reported as a naming limitation, not archive incompatibility.

- Synchronizes a saved DBC into a selected server `data\\dbc` directory with backup.
